## สถาปัตยกรรมหลัก (Software Architecture)

การพัฒนา Software ที่ดีและมีประสิทธิภาพควรประกอบไปด้วย 3 ส่วนหลัก ได้แก่ Frontend (ส่วนแสดงผลและโต้ตอบกับผู้ใช้งาน), Backend (ส่วนจัดการลอจิกและประมวลผล) และ Database (ส่วนการจัดเก็บและคิวรีข้อมูล)

ก่อนที่จะเริ่มพัฒนาระบบ ควรมีการเลือก Architecture ให้เหมาะสมกับโจทย์ของงาน โดยคำนึงถึงเป้าหมายหลัก เช่น:

| เป้าหมาย | สิ่งที่ควรเน้น |
|---|---|
| ตอบสนองรวดเร็ว | Performance / Caching |
| ทนทาน ไม่ล่มทั้งระบบ | Resilience / Availability |
| ปลอดภัยสูง | Security Layer / Isolation |

### รูปแบบสถาปัตยกรรม

1. **Monolithic** — เขียนโค้ดและทำงานทุกอย่างรวมไว้ในที่เดียว (Frontend, Backend, Database) เริ่มต้นง่าย แต่ scale ยาก
2. **Layered Architecture** — แบ่งแยก Layer ให้ชัดเจนขึ้น แต่โดยรวมยังคงอยู่ด้วยกัน
3. **Client-Server** — แยกส่วน Server และ Client อย่างชัดเจน สื่อสารกันผ่านเครือข่าย แบ่งได้เป็น 2 ลักษณะ:
   - **Stateless** — Server ไม่จดจำ Session เดิมของการเข้าใช้งาน
   - **Stateful** — Server จดจำสถานะของผู้ใช้ และเชื่อมต่อตลอดเวลา
4. **N-Tier Architecture** — พัฒนาต่อยอดจาก Layered โดยแยกแต่ละ Tier ไปอยู่บนเซิร์ฟเวอร์คนละเครื่องอย่างชัดเจน

## การทดสอบที่ 2: Service Availability & Failover

### ปัญหาที่พบ
จากการจำลองสถานการณ์ Server ล่มด้วยคำสั่ง `docker stop` พบว่าระบบเดิมเกิดปัญหา **Service Disruption** โดยแสดงผลเป็นหน้า Error `502 Bad Gateway` หรือ `504 Gateway Timeout` เนื่องจาก Nginx พยายามเรียกไปยัง Instance ที่ใช้งานไม่ได้แล้ว และไม่มีกลไกสลับไปยังตัวที่ยังทำงานอยู่โดยอัตโนมัติ

### แนวทางการแก้ไข
ปรับปรุงการตั้งค่าใน `nginx/default.conf` เพื่อเพิ่มความทนทาน (Resilience):

1. **Dynamic Service Discovery** — ใช้คำสั่ง `resolver` เพื่อให้ Nginx คอยอัปเดต IP Address ของ Service จาก Docker DNS อย่างสม่ำเสมอ
2. **Automatic Failover** — เพิ่มคอนฟิก `proxy_next_upstream` เพื่อให้ระบบสลับไปเรียก App Instance ตัวถัดไปทันทีหากตัวปัจจุบันตอบสนองผิดปกติ

### ผลการทดสอบ
หลังการแก้ไข ระบบสามารถให้บริการได้อย่างต่อเนื่อง (High Availability) แม้จะมี Instance บางส่วนล่มไป ผู้ใช้งานฝั่ง Frontend จะไม่พบหน้า Error และสามารถใช้งานได้ตามปกติ

## การทดสอบที่ 3: Horizontal Scaling & Performance

ทดสอบประสิทธิภาพการรับโหลดด้วยเครื่องมือ `wrk`:

```bash
wrk -t4 -c50 -d20s http://localhost/api/tasks
```

### ผลเปรียบเทียบ

| Metric | Case A: 1 Instance (Baseline) | Case B: 3 Instances (Scale-out) |
|:---|:---:|:---:|
| **Avg Latency** | 31.42 ms | **24.87 ms** |
| **Throughput (Req/s)** | 2,741.60 | **3,489.33** |

- **Latency** ลดลงประมาณ 21% จากการกระจายโหลด
- **Throughput** เพิ่มขึ้นอย่างเห็นได้ชัดจาก Round-Robin Load Balancing

## สรุปองค์ความรู้ที่ได้รับ

### กลยุทธ์การเพิ่ม Tier ในระบบ

- **เพิ่ม Cache Tier (Redis)** — เมื่อต้องการลดภาระ Database (DB Bottleneck) และเพิ่มความเร็วในการอ่านข้อมูล
- **เพิ่ม Load Balancer Tier (Nginx)** — เมื่อต้องการขยายขีดความสามารถ (Scaling) ให้รองรับผู้ใช้จำนวนมากพร้อมกัน

### ข้อควรระวัง

ควรระวังเรื่อง **Over-engineering** หากระบบยังมีขนาดเล็กและมีผู้ใช้งานไม่มาก การเพิ่ม Tier ที่ซับซ้อนเกินไปอาจทำให้การบำรุงรักษาทำได้ยากและเพิ่มต้นทุนโดยไม่จำเป็น

> หลักการสำคัญ: "เพิ่ม Tier เมื่อมีปัญหาจริง ไม่ใช่เพราะ 'น่าจะดี'"
